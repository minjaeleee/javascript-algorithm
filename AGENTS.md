# JavaScript Algorithm 학습 사이트 작업 지침

## 프로젝트 목표

- JavaScript 알고리즘 강의 내용을 학습자가 직접 정리하는 저장소다.
- 정리한 글을 블로그처럼 탐색하고 읽을 수 있는 정적 웹사이트로 제공한다.
- 데스크톱과 모바일 모두에서 읽기 쉽고 기능이 안정적으로 동작해야 한다.
- GitHub 저장소에서 관리하고 GitHub Pages로 배포한다.
- 향후 사용자가 학습 내용이나 목차 변경을 전달하면, 구현뿐 아니라 글 작성 방식과 학습 순서도 함께 안내한다.

## 현재 단계

1. `javascript-algorithm` 저장소 생성 완료
2. VitePress 기술 구성 결정 및 기본 사이트 구현 완료
3. 목차와 문서 경로의 초기 구조 구성 완료
4. GitHub Actions 기반 GitHub Pages 자동 배포 구성 완료
5. 첫 번째 학습 문서를 작성하며 공통 문서 템플릿 검증
6. 모바일·데스크톱 표시와 실제 배포 검증

## 기술 구성

- 정적 사이트 생성기로 VitePress를 사용한다.
- 콘텐츠는 `docs/` 아래의 Markdown으로 작성하여 알고리즘 설명과 JavaScript 코드를 관리한다.
- VitePress 설정은 `docs/.vitepress/config.mts`, 테마 확장은 `docs/.vitepress/theme/`에서 관리한다.
- GitHub Pages의 프로젝트 경로를 지원하기 위해 VitePress `base`는 `/javascript-algorithm/`로 유지한다.
- `main` 브랜치 변경 시 `.github/workflows/deploy.yml`에서 빌드하고 GitHub Pages에 배포한다.
- 모바일 우선 반응형 UI, 접근성, 빠른 로딩, 코드 가독성을 우선한다.
- 학습 저장소이므로 불필요한 서버, 데이터베이스, 인증 기능은 도입하지 않는다.

## 사이트에 필요한 기본 기능

- 장/절 단위 사이드바 또는 모바일 목차 메뉴
- 이전 글·다음 글 이동
- 현재 문서의 제목과 소제목 탐색
- JavaScript 코드 문법 강조와 코드 복사
- 모바일에서 가로 스크롤이 가능한 코드 블록
- 읽기 편한 본문 폭과 타이포그래피
- GitHub Pages 새로고침 및 직접 URL 접근 호환성
- 추후 필요할 때 검색, 다크 모드, 학습 진행 표시를 단계적으로 검토

## 콘텐츠 작성 가이드

각 개념 문서는 가능한 한 다음 순서를 따른다.

1. 학습 목표
2. 핵심 개념
3. 동작 원리 또는 시각적 예시
4. JavaScript 구현
5. 시간 복잡도와 공간 복잡도
6. 자주 하는 실수와 경계 조건
7. 짧은 예제
8. 복습 질문

각 문제 풀이 문서는 가능한 한 다음 순서를 따른다.

1. 문제 요약과 출처 링크
2. 입력·출력 및 제약 조건
3. 접근 방법
4. 알고리즘 선택 근거
5. JavaScript 풀이
6. 복잡도 분석
7. 다른 풀이 또는 개선점
8. 배운 점

강의 문장을 그대로 옮기기보다 직접 이해한 표현으로 요약한다. 외부 문제, 이미지, 인용문은 출처와 링크를 표시하고 저작권을 준수한다. 실행 가능한 코드는 필요한 경우 테스트 케이스와 함께 둔다.

## 콘텐츠 디렉터리 및 이름 규칙

- 대단원은 사이트 목차 순서에 따라 `01-data-structures`처럼 1부터 시작하는 두 자리 번호와 영문 슬러그를 사용한다.
- 강의에서 전달된 기존 장 번호를 사이트의 목차 번호나 파일 경로에 그대로 사용하지 않는다.
- 문서는 `01-data-structure.md`처럼 정렬 가능한 두 자리 번호를 사용한다.
- 화면에 표시되는 제목은 자연스러운 한국어와 필요한 영문 용어를 함께 쓴다.
- URL과 파일명은 공백 없는 소문자 kebab-case를 사용한다.
- 문제 풀이 제목의 임시 표현인 `문제 풀이 1` 등은 실제 문제명이 정해지면 교체한다.

## 목차 확인 사항

- 강의 원본의 장 번호와 관계없이 사이트의 첫 번째 대단원은 `1.`부터 시작하고 이후 대단원도 순서대로 번호를 매긴다.
- 3, 5, 6, 8번 대단원의 `3문제 1~N` 표기는 강의의 실제 제목을 받으면 수정한다.
- 10번과 11번 대단원의 2번·3번 제목이 중복되어 있으므로 실제 문제명 또는 구분명을 받으면 수정한다.
- 새 목차를 받을 때 번호, 표시 제목, 파일 슬러그, 선행 학습 관계를 함께 갱신한다.

## 구현 품질 기준

- 사용자 콘텐츠와 기존 변경 사항을 임의로 삭제하거나 덮어쓰지 않는다.
- 모든 페이지가 작은 모바일 화면과 일반 데스크톱 화면에서 읽을 수 있어야 한다.
- 의미 있는 HTML 구조, 키보드 탐색, 충분한 색상 대비를 적용한다.
- 코드와 설정 변경 후 빌드 및 관련 검사를 실행한다.
- GitHub Pages 배포 설정에서는 저장소 기본 경로와 정적 자산 경로를 검증한다.
- 기능과 구조가 바뀌면 README와 이 문서를 함께 최신화한다.

## 협업 방식

- 사용자가 강의 내용을 전달하면 문서 구조, 핵심 설명, 예제 코드, 복잡도 분석, 복습 포인트를 함께 제안한다.
- 정보가 부족하더라도 안전한 기본값으로 초안을 만들고, 학습 내용이나 공개 범위를 바꾸는 중요한 결정만 확인한다.
- 한 번에 큰 범위를 만들기보다 한 장의 구조와 품질을 먼저 확정한 뒤 같은 패턴으로 확장한다.
